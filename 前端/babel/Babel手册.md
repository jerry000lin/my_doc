# Babel手册

## Babel 的前世今生

Babel 是一个 JavaScript 编译工具，核心作用是将一份源码按规则转换成另一份源码。

它最早诞生于 2014 年，最初名称为 6to5，主要解决的问题是将 ES6 代码转换为 ES5，使代码可以在旧浏览器中运行。当时 ES6 引入了类、箭头函数、模板字符串等新语法，显著提升了开发效率，但浏览器支持并不统一，这使得 Babel 快速流行起来。

在这一阶段，Babel 的核心价值是：允许开发者使用新语法，同时保证旧环境兼容。

随着前端工程化发展，JavaScript 不再只是浏览器直接执行的脚本语言，而是逐步进入“先编译再运行”的模式。开发中开始出现更多需要转换的语法形态，例如 JSX、TypeScript 等。Babel 在这一阶段承担了统一的源码转换角色，使这些语法可以在工程中被使用。

到了 2026 年，Babel 的使用方式发生了变化：

* 默认前端项目（如 Vite）很多已经不再依赖 Babel 作为主编译工具
* 高性能编译器（如 SWC、Oxc）更多承担基础编译职责
* Babel 更多出现在：

  * 需要自定义代码转换的场景
  * 使用特定插件（如 React Compiler）的场景
  * 老项目或复杂构建链路中

---

## Babel 如何解决兼容问题

Babel 处理兼容问题主要有两种方式：transform 和 polyfill。

transform 用来解决语法问题。当代码使用了较新的语法，而目标环境无法解析时，Babel 会把这些语法改写为旧环境可以执行的形式，例如把箭头函数转换为普通函数。

polyfill 用来解决运行时能力问题。当目标环境缺少某种能力时，需要额外补充实现，例如旧环境中没有 `Promise`。

可以简单理解为：

* transform：改写代码结构
* polyfill：补充运行能力

---

## Babel 的编译流程

Babel 的核心流程可以概括为三步。

首先是解析。源码会被转换为 AST（抽象语法树）。

然后是转换。Babel 会遍历 AST，根据规则修改节点结构。

最后是生成。修改后的 AST 会被重新生成代码。

整个流程可以概括为：

**源代码 → AST → 修改 AST → 新代码**

---

## Babel 的核心机制：Plugin 与 Preset

Babel 的能力是通过插件机制实现的。

plugin 是最小的转换单元，每个 plugin 负责一类具体的代码转换逻辑。

preset 是一组 plugin 的集合，用于减少配置复杂度。

可以这样理解：

* plugin：单个功能
* preset：功能组合

其中最重要的 preset 是 `@babel/preset-env`。

它的作用是：

**根据你配置的目标浏览器或运行环境，自动决定需要做哪些语法转换。**

这个目标环境通常通过 **browserslist** 指定，例如：

```txt
> 0.25%
not dead
```

Babel 会根据这份配置判断：

* 已支持的语法 → 不转换
* 不支持的语法 → 才转换

因此你不需要手动配置一堆转换规则，而是只需要指定：
**我要支持哪些环境**

---

## 总结

Babel 是一个 JavaScript 编译工具，负责源码转换。

它最初用于解决 ES6 到 ES5 的兼容问题，后来发展为前端工程化中的通用转换工具。

Babel 解决兼容问题的方式：

* transform：语法转换
* polyfill：运行时能力补齐

Babel 的工作流程：

**Parsing → Transformation → Generation**

Babel 的核心结构：

* plugin：最小转换单元
* preset：插件集合
* preset-env：根据 browserslist 自动决定转换策略
